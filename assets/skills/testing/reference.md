# 软件测试基础能力 - 详细参考

本文档提供测试基础能力的详细说明、案例分析和应用指南。

## 方法概述

测试用例设计方法可分为三大类：

| 方法类别 | 适用方法 | 主要特点 |
|---------|---------|----------|
| 黑盒测试 | 等价类、边界值、因果图、判定表、正交试验、场景法 | 关注外部行为，不涉及内部代码 |
| 白盒测试 | 语句覆盖、分支覆盖、路径覆盖等 | 关注代码结构和逻辑 |
| 基于经验的测试 | 错误推测法、探索性测试 | 依赖测试人员经验与直觉 |

### 检出率对比

| 方法 | 适用阶段 | 典型检出率 |
|------|----------|-----------|
| 等价类划分法 | 单元/系统 | 25%-30% |
| 边界值分析法 | 单元/系统 | 38%（单元） |
| 判定表法 | 系统/集成 | 41%（系统） |
| 场景法 | 系统/验收 | 52%（探索性） |
| 错误推测法 | 探索性/回归 | 依赖经验 |

## 等价类划分法

### 详细步骤

1. **确定输入条件**：从需求规格说明中识别所有可能的输入条件
2. **划分等价类**：为每个输入条件划分有效等价类和无效等价类
3. **建立等价类表**：将所有等价类整理成表格
4. **设计测试用例**：为每个等价类设计至少一个测试用例

### 案例分析：用户名注册

**需求**：
- 必填
- 长度为 3-15 个字符
- 仅允许英文字母、数字、中划线(-)、下划线(_)
- 首字符必须为字母或数字

**等价类划分**：

| 输入条件 | 有效等价类 | 编号 | 无效等价类 | 编号 |
|----------|-----------|-----|------------|-----|
| 是否填写 | 已填写 | (1) | 为空 | (4) |
| 长度 | 3-15字符 | (2) | 长度<3（如"ab"） | (5) |
|        |           |     | 长度>15（如"abcdefghijklmnopq"） | (6) |
| 字符类型 | 字母、数字、-、_ 组合 | (3) | 包含非法字符（如"user@name"） | (7) |

**测试用例设计**：
- 用例1：`Tom123` → 覆盖(1)(2)(3)
- 用例2：空输入 → 覆盖(4)
- 用例3：`ab` → 覆盖(5)
- 用例4：`abcdefghijklmnopq` → 覆盖(6)
- 用例5：`user@name` → 覆盖(7)

**关键原则**：
- 设计新用例尽可能多地覆盖尚未被覆盖的有效等价类
- 每个新用例仅覆盖一个尚未被覆盖的无效等价类，避免错误遮掩

### 组合等价类

当多个输入条件相互影响时，需要考虑组合情况。

### 能力提升

- **初级**：能识别简单输入的等价类
- **中级**：能处理多个输入条件的等价类组合
- **高级**：能利用等价类关系优化测试用例设计

## 边界值分析法

### 详细步骤

1. **确定输入边界**：从需求规格中识别输入域的上限、下限
2. **选择测试点**：选取边界值、边界±1 值
3. **考虑输出边界**：分析输出结果的边界条件
4. **设计预期结果**：明确每个测试用例的预期结果

### 测试点选取规则

对于闭区间 [a, b]：
- **上点**：边界上的精确值（a, b）
- **离点**：紧邻边界的偏差值（a-1, b+1）
- **内点**：边界范围内的任意普通值

**标准六点法**：
- 最小值-1（离点）
- 最小值（上点）
- 最小值+1（内点）
- 最大值-1（内点）
- 最大值（上点）
- 最大值+1（离点）

### 理论依据

约 60% 的软件错误集中在边界条件处理环节，因此边界值测试是发现缺陷的有效方法。

### 案例分析：年龄输入

**需求**：用户年龄范围 18-60 岁

**测试用例**：
- 17 岁（边界下限-1，离点，无效）→ 应报错
- 18 岁（边界下限，上点，有效）→ 允许注册
- 19 岁（边界下限+1，内点，有效）→ 允许注册
- 59 岁（边界上限-1，内点，有效）→ 允许注册
- 60 岁（边界上限，上点，有效）→ 允许注册
- 61 岁（边界上限+1，离点，无效）→ 应报错

### 边界点类型

| 类型 | 描述 | 示例 |
|------|------|------|
| 上点 | 边界上的精确值（闭区间包含） | 年龄 18 岁 |
| 离点 | 紧邻边界的偏差值 | 年龄 17 岁 |
| 内点 | 边界范围内的任意普通值 | 年龄 30 岁 |

### 多维度边界

当存在多个边界条件时，需要考虑边界组合。

### 能力提升

- **初级**：能识别单一维度的边界
- **中级**：能处理多维度边界组合
- **高级**：能预测潜在边界缺陷模式（如溢出、精度问题）

## 场景法

### 详细步骤

1. **分析业务流程**：识别核心业务流程和关键节点
2. **绘制流程图**：明确基本流（正常流程）、备选流（分支路径）、异常流（失败路径）
3. **识别风险点**：找出流程中的决策点和异常分支
4. **设计测试场景**：为基本流和每个备选流设计测试用例
5. **组合场景**：将基本流与各备选流组合，形成独立测试场景

### 流程类型

| 流程类型 | 描述 | 示例 |
|---------|------|------|
| 基本流 | 主路径，所有输入合法，系统正常响应 | 浏览→加购→下单→支付→完成 |
| 备选流 | 分支路径，条件判断后的不同结果 | 不同支付方式选择 |
| 异常流 | 失败路径，输入错误或系统异常 | 库存不足、支付失败 |

### 案例分析：电商下单流程

**基本流**：浏览商品 → 加入购物车 → 下单 → 支付 → 订单完成

**备选流与异常流**：
- 商品库存不足 → 提示缺货
- 购物车商品失效 → 移除失效商品
- 收货地址无效 → 提示修改地址
- 支付超时 → 订单自动取消
- 支付失败 → 提示重新支付
- 支付后库存更新失败 → 订单异常处理

**测试场景示例**：
1. 正常下单流程（基本流）
2. 下单后库存不足（异常流）
3. 下单后支付超时自动取消（异常流）
4. 下单后取消订单（备选流）

### 适用场景

- 电商购物流程
- 金融交易
- 社交互动
- 多个功能点串联的复杂系统

### 核心思想

"用户怎么用，我们就怎么测"——模拟用户真实操作流程，发现集成缺陷、业务逻辑错误和数据一致性问题。

### 能力提升

- **初级**：能绘制简单业务流程图
- **中级**：能设计包含基本流和备选流的完整场景
- **高级**：能识别流程中的关键风险点并设计针对性测试

## 因果图法

### 基本概念

因果图法是一种图形化工具，用于表示输入条件（原因）与输出结果（结果）之间的逻辑关系，以及输入之间或输出之间的约束条件。特别适用于多条件组合且存在复杂制约关系的功能模块。

### 基本逻辑符号

| 符号 | 名称 | 逻辑 | 说明 |
|------|------|------|------|
| 恒等 | Identity | 若 c=1 则 e=1 | 原因直接导致结果 |
| 非 | NOT | 若 c=1 则 e=0 | 原因取反导致结果 |
| 或 | OR | c1 或 c2 或 ... 任一为 1 则 e=1 | 任一原因即可触发结果 |
| 与 | AND | c1 且 c2 且 ... 全为 1 则 e=1 | 所有原因同时满足才触发结果 |

### 约束符号

| 约束 | 符号 | 说明 |
|------|------|------|
| 互斥 | E（Exclusive） | 最多只有一个条件为真 |
| 包含 | I（Inclusive） | 至少有一个条件为真 |
| 唯一 | O（Only One） | 有且只有一个条件为真 |
| 要求 | R（Require） | 若 a 为真则 b 必须为真 |
| 屏蔽 | M（Mask） | 若 a 为真则 b 必须为假 |

### 使用步骤

1. **分析需求**：分解复杂功能，提取关键输入与输出
2. **确定原因与结果**：将输入条件定义为"原因"（c1, c2...），输出定义为"结果"（e1, e2...）
3. **建立因果关系**：使用逻辑符号连接原因与结果
4. **添加约束条件**：标注约束，排除无效组合
5. **转换为判定表**：将因果图转化为判定表，列出所有可能的状态组合
6. **生成测试用例**：将判定表中每一列对应一个有效测试用例

### 案例分析：登录功能

**需求**：用户名正确且密码正确 → 登录成功

- **原因**：
  - c1: 用户名正确
  - c2: 密码正确
- **结果**：
  - e1: 登录成功
- **逻辑关系**：c1 ∧ c2 → e1

**判定表**：

| 规则 | c1 | c2 | e1 |
|-----|----|----|----|
| 1 | T | T | 成功 |
| 2 | T | F | 失败 |
| 3 | F | T | 失败 |
| 4 | F | F | 失败 |

### 适用场景

- 登录控制
- 支付流程
- 审批流程
- 多条件组合的复杂规则

### 能力提升

- **初级**：能识别简单的因果关系
- **中级**：能绘制基本因果图并转换为判定表
- **高级**：能处理复杂的多层因果关系和约束条件

## 判定表法

### 基本概念

判定表法是一种结构化的测试用例设计方法，用于描述多条件组合下的预期行为。它以表格形式清晰展示"条件—动作"映射关系，帮助测试人员穷举所有合法的条件组合，确保测试覆盖的完整性与无遗漏性。

### 判定表构成

| 部分 | 说明 |
|------|------|
| 条件桩 | 列出所有输入条件 |
| 动作桩 | 列出所有可能的操作 |
| 条件项 | 每种条件的取值组合 |
| 动作项 | 对应的动作结果 |

**规则数量**：若有 n 个布尔条件，理论上存在 $2^n$ 条规则。

### 使用步骤

1. **识别条件和动作**：分析需求文档，列出所有影响系统行为的输入条件和系统可能执行的动作
2. **确定规则数量**：若 n 个条件均为布尔型，则初始规则数为 $2^n$
3. **填写条件项**：枚举所有条件组合，常用二进制递增法生成
4. **填写动作项**：根据每种条件组合，填写对应的应执行动作
5. **简化判定表**：合并具有相同动作的规则，使用"-"标记无关条件，提高效率
6. **生成测试用例**：将每条规则转化为一个测试用例，明确输入数据和预期输出

### 案例分析：会员折扣计算

**需求规则**：
- 普通会员：订单金额≥100元享9折
- 金牌会员：订单金额≥100元享8折
- 钻石会员：订单金额≥100元享7折

**判定表**：

| 规则编号 | 会员等级 | 订单金额≥100元 | 折扣率 |
|---------|----------|----------------|--------|
| 1 | 普通 | 是 | 9折 |
| 2 | 普通 | 否 | 原价 |
| 3 | 金牌 | 是 | 8折 |
| 4 | 金牌 | 否 | 原价 |
| 5 | 钻石 | 是 | 7折 |
| 6 | 钻石 | 否 | 原价 |

**简化判定表示例**：

| 规则 | C1: VIP用户 | C2: 金额≥1000 | A1: 9折 | A2: 原价 |
|-----|------------|-------------|---------|---------|
| 1 | T | T | X |   |
| 2 | T | F |   | X |
| 3 | F | - |   | X |

注："X" 表示执行该动作，"-" 表示无关

### 适用场景

- 折扣计算
- 审批流程
- 业务规则复杂的系统
- 多条件组合的决策逻辑

### 能力提升

- **初级**：能编写简单判定表
- **中级**：能优化和简化复杂判定表
- **高级**：能自动生成并优化大型判定表

## 正交试验法

### 基本概念

正交试验法是一种利用正交表从多因素多水平组合中挑选代表性子集进行测试的方法，旨在解决"组合爆炸"问题。具备"均匀分散、齐整可比"的特点，能在有限资源下实现高效的两两交互覆盖。

### 正交表表示

正交表记作 $L_n(t^c)$，其中：
- $L$：正交表代号
- $n$：试验次数（行数）
- $t$：每个因素的水平数
- $c$：最多可安排的因素个数（列数）

**示例**：$L_9(3^4)$ 表示需做 9 次试验，最多可考察 4 个因素，每个因素有 3 个水平。

### 常用正交表

| 正交表 | 因素数 | 最大水平 | 测试用例数 |
|--------|--------|----------|-----------|
| L4(2^3) | 3 | 2 | 4 |
| L8(2^7) | 7 | 2 | 8 |
| L9(3^4) | 4 | 3 | 9 |
| L16(4^5) | 5 | 4 | 16 |

### 使用步骤

1. **明确试验目的**：确定要优化的目标，如转化率、稳定性等
2. **挑选因素，确定水平**：分析影响结果的关键因素及其可能取值
3. **选择合适的正交表**：原则是正交表的列数 ≥ 因素数，水平数 ≥ 各因素的最大水平数
4. **明确实验方案**：将因素与水平映射到正交表的列上，生成具体测试用例
5. **执行测试并获取结果**：按计划执行并记录结果
6. **补充边界与异常用例**：正交表主要覆盖主效应和两两交互，需额外添加边界值、错误猜测等用例

### 案例分析：浏览器兼容性测试

**因素和水平**：
- 浏览器：Chrome、Firefox、Safari（3 水平）
- 操作系统：Windows、macOS、Linux（3 水平）

**使用 L9(3^4) 正交表**：

| 用例编号 | 浏览器 | 操作系统 |
|----------|--------|----------|
| 1 | Chrome | Windows |
| 2 | Chrome | macOS |
| 3 | Chrome | Linux |
| 4 | Firefox | Linux |
| 5 | Firefox | Windows |
| 6 | Firefox | macOS |
| 7 | Safari | macOS |
| 8 | Safari | Linux |
| 9 | Safari | Windows |

**覆盖率分析**：相比全量测试的 9 种组合，已实现高效覆盖。

### 适用场景

- 兼容性测试
- 配置项组合测试
- 解决组合爆炸问题
- 多因素多水平的测试场景

### 能力提升

- **初级**：能使用基础正交表设计测试用例
- **中级**：能根据因素水平选择合适的正交表
- **高级**：能设计混合正交表和特殊场景测试方案

## 错误推测法

### 基本概念

错误推测法是一种基于测试人员经验、直觉和对系统潜在缺陷认知的黑盒测试方法。其核心是列举程序中常见错误模式和易错点，有针对性地设计测试用例。

**核心思想**："站在破坏者的角度思考"——从"证明它是对的"转变为"想办法证明它是错的"。

### 使用步骤

1. **理解与分析需求**：彻底掌握功能规格说明书和业务逻辑
2. **分析被测对象**：了解系统架构、技术栈、历史缺陷分布，识别薄弱环节
3. **收集经验与知识库调用**：回顾过往项目中的缺陷报告、常见错误清单，组织头脑风暴会议集思广益
4. **推测潜在错误点**：列举可能出错的场景，如边界条件、异常输入、并发冲突等
5. **设计针对性测试用例**：针对每个推测的错误设计具体的测试步骤、输入数据和预期结果
6. **优先级排序与执行**：根据风险程度和影响范围对测试用例进行优先级排序，优先执行高风险用例
7. **结合其他方法使用**：与等价类、边界值等方法结合，提高整体覆盖率

### 思考维度

| 维度 | 说明 | 示例 |
|------|------|------|
| 历史缺陷 | 基于过往缺陷推测类似问题 | 类似功能的已知缺陷 |
| 边界情况 | 边界值法未覆盖的边缘场景 | 非常规边界值 |
| 异常输入 | 非常规的用户输入 | SQL注入、XSS攻击 |
| 资源约束 | 内存、磁盘空间、网络限制 | 磁盘空间不足 |
| 并发冲突 | 多用户同时操作的场景 | 抢购、并发提交 |
| 时序问题 | 依赖执行顺序的逻辑 | 操作顺序影响结果 |

### 案例分析：文件上传功能

**常见易错点**：
- 上传 0 字节空文件
- 上传超大文件（如 5GB）
- 上传病毒文件
- 文件名含特殊字符或超长字符串
- 扩展名伪装（如 .exe 改为 .jpg）
- 并发上传相同文件
- 上传后立即删除再上传
- 上传中网络中断

**测试用例示例**：
- 用例1：上传 0 字节文件 → 应拒绝或正确处理
- 用例2：上传超过限制大小的文件 → 应提示文件过大
- 用例3：上传包含恶意脚本的文件 → 应进行安全检查
- 用例4：上传文件名包含特殊字符的文件 → 应正确处理或拒绝
- 用例5：上传扩展名伪装的文件 → 应验证真实文件类型

### 错误推测模式库

建立常见缺陷模式库，包括：
- **输入验证缺陷**：SQL注入、XSS、命令注入
- **边界处理缺陷**：off-by-one错误、溢出
- **状态管理缺陷**：状态不一致、死锁
- **并发控制缺陷**：竞态条件、资源冲突
- **权限控制缺陷**：越权访问、权限绕过
- **性能相关缺陷**：内存泄漏、资源耗尽

### 适用场景

- 时间紧迫项目
- 探索性测试阶段

## 方法组合应用

### 组合策略

1. **等价类 + 边界值**：常规测试用例设计，覆盖输入域和边界条件
2. **场景法 + 错误推测**：业务流程测试，模拟真实用户行为
3. **因果图 + 判定表**：复杂规则测试，处理多条件组合
4. **正交试验 + 配置测试**：多因素组合测试，解决组合爆炸
5. **探索性测试 + 其他方法**：补充覆盖遗漏场景

### 综合应用策略

| 测试阶段 | 推荐方法 | 说明 |
|---------|---------|------|
| 单元测试 | 等价类、边界值 | 验证函数逻辑，辅以条件组合覆盖 |
| 集成测试 | 判定表、状态转换法 | 验证接口逻辑和数据流 |
| 系统测试 | 场景法、正交试验 | 统筹全局，覆盖配置组合 |
| 回归测试 | 核心用例 + 探索性测试 + 错误推测 | 保证稳定性，发现新问题 |

### 决策树建议

- **功能逻辑简单** → 优先使用等价类 + 边界值
- **多条件组合** → 引入判定表或因果图
- **多平台兼容性测试** → 采用正交试验法
- **用户旅程验证** → 以场景法为主
- **所有方法之后** → 用错误推测法查漏补缺

### 案例分析：电商平台下单功能测试

**1. 使用等价类划分法验证字段输入**
- 订单金额：有效（0.01-99999.99），无效（<0.01, >99999.99, 非数字）
- 收货地址：有效（合法格式），无效（空、超长、非法字符）

**2. 使用边界值分析法测试价格极限**
- 0.01 元（最小值）
- 99999.99 元（最大值）
- 0 元（边界-1，无效）
- 100000 元（边界+1，无效）

**3. 使用判定表法处理优惠券叠加规则**
| 会员等级 | 订单金额≥100 | 有优惠券 | 最终折扣 |
|---------|-------------|---------|---------|
| 普通 | 是 | 是 | 8.5折 |
| 普通 | 是 | 否 | 9折 |
| 普通 | 否 | - | 原价 |
| VIP | 是 | 是 | 7折 |
| VIP | 是 | 否 | 8折 |
| VIP | 否 | - | 原价 |

**4. 使用场景法模拟全流程**
- 正常购买：浏览→加购→下单→支付→完成
- 库存不足：浏览→加购→下单→提示缺货
- 支付失败：浏览→加购→下单→支付失败→重新支付

**5. 使用错误推测法尝试异常行为**
- 并发抢购
- 重复提交订单
- 支付后修改地址
- 退款后再次下单

## 方法对比总结

| 方法 | 适用阶段 | 主要优点 | 主要缺点 | 典型检出率 |
|------|----------|----------|----------|-----------|
| 等价类划分法 | 单元/系统 | 减少冗余用例，提升效率 | 无法覆盖边界问题 | 25%-30% |
| 边界值分析法 | 单元/系统 | 高效发现临界错误 | 不适用于非数值输入 | 38%（单元） |
| 判定表法 | 系统/集成 | 全面覆盖复杂逻辑 | 条件过多时指数膨胀 | 41%（系统） |
| 正交试验法 | 系统/兼容性 | 解决组合爆炸，高效覆盖 | 高阶交互覆盖有限 | 适中 |
| 场景法 | 系统/验收 | 用户视角，验证端到端流程 | 可能忽略底层细节 | 52%（探索性） |
| 错误推测法 | 探索性/回归 | 快速发现隐藏缺陷 | 主观性强，难以量化 | 依赖经验 |
